# 联系与反馈

默认使用 GitHub 作为协作入口，让产品问题、缺陷和贡献上下文对社区可见、可追踪。

| 需求 | 去哪里 | 需要包含 |
| --- | --- | --- |
| 反馈缺陷或异常工作流 | [GitHub Issues](https://github.com/datagallery-lab/datafoundry/issues) | 版本、环境、复现步骤、预期行为、实际行为和相关日志。 |
| 提出功能或数据源建议 | [GitHub Issues](https://github.com/datagallery-lab/datafoundry/issues) | 场景、数据源类型、用户角色、预期结果，以及当前工作流为什么不足。 |
| 加入交流群 | [README 社区与交流](https://github.com/datagallery-lab/datafoundry/blob/main/README_zh.md#-%E7%A4%BE%E5%8C%BA%E4%B8%8E%E4%BA%A4%E6%B5%81) | 扫描 README 中的二维码，讨论产品使用、路线图和落地场景。 |
| 改进文档 | [Pull requests](https://github.com/datagallery-lab/datafoundry/pulls) | 页面路径、建议措辞、视觉变更截图和验证步骤。 |
| 贡献代码 | [参与贡献](contributing.md) | 范围、关联 issue、测试或 smoke 检查，以及迁移或兼容性说明。 |
| 分享场景案例 | [场景案例](scenario-cases.md) | 业务背景、示例 schema 或数据形态、预期分析路径，以及可公开展示的边界。 |
| 安全或数据暴露问题 | [GitHub Security Advisories](https://github.com/datagallery-lab/datafoundry/security/advisories) | 脱敏后的影响描述和涉及范围。不要在初始报告中包含密钥、凭据或私有数据。 |

## 提 issue 之前

- 先搜索现有 issue，避免重复反馈。
- 尽量确认问题是否能在最新 `main` 分支复现。
- 提供足够上下文，让维护者不依赖私有系统也能复现问题。

公开示例中的截图和日志需要先脱敏。DataFoundry 本身强调可治理的数据工作流，社区反馈也应遵循同样的边界。
